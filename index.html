시작하기 — 기존 환경 흡수

설치 & 첫 실행 (3단계)
① 설치: curl -fsSL https://omp.sh/install | sh (macOS·Linux·WSL) — 또는 bun install -g @oh-my-pi/pi-coding-agent (bun ≥ 1.3.14 · brew·Windows도 지원)
② 로그인: omp → /login으로 기존 Claude/ChatGPT 등 구독 OAuth 그대로 — 별도 API 키 불필요
③ 사용: 프로젝트에서 omp → 기존 CLAUDE.md·AGENTS.md 자동 인식. 끝.
상호운용 — “갈아타기 비용 낮음”
읽어 합치는 것: ~/.claude·~/.codex·~/.gemini의 설정·지침 파일(CLAUDE.md 등)·슬래시 명령·훅·스킬.
우선순위(충돌 시): native(.omp) 100 › claude 80 › codex 70 › gemini 60.
비파괴(read-only) — 읽어 합칠 뿐 수정 안 함 → Claude Code·Codex 계속 병행해도 안전.
기존 워크플로를 버리지 않고 그대로 흡수 → 시도 비용이 거의 0이고 되돌리기 쉽다.
출처 omp.sh/install(README) · omp 스킬 omp-pitch §5 · 내장문서 config-usage(우선순위 native 100›claude 80›codex 70›gemini 60).

참고 · 외부 설정 흡수 제어

외부 설정 흡수 끄기

omp는 ~/.claude·~/.codex의 슬래시·스킬·훅을 읽어 합친다 — 갈아타기 비용을 낮추는 핵심이다. 다만 흡수된 스킬 이름·설명이 시스템 프롬프트에 상시 로드, 슬래시 메뉴 비대 등이 신경 쓰여 흡수를 원치 않으면, omp 안에서 끌 수 있다 — OS 분리·삭제 불필요. 이게 가장 가벼운 해법이다. 단 이 부담은 흡수량에 비례한다 — 적게 깔렸으면 체감은 미미하다.

~/.omp/agent/config.yml
commands:
  enableClaudeUser: false
  enableClaudeProject: false
skills:
  enableClaudeUser: false
  enableCodexUser: false
  enableClaudeProject: false
흡수원별 토글
위 enable*를 각각 false로 — 네이티브 스킬은 enablePi*로 유지. 흡수만 골라 끄는 단일 스위치는 없다. 스킬 전체는 skills.enabled=false·CLI --no-skills로 한 번에(단 네이티브까지 꺼짐). codex 슬래시는 전용 토글이 없어 ~/.codex/commands를 비워야 막힌다. /settings에서도 조정.
일부만
특정 스킬만 빼기 ignoredSkills(글롭) · 특정만 허용 includeSkills.
판단 — 흡수는 기본 켜짐이고 그게 덱의 핵심 셀링포인트(이주 편의)다. 끄면 그 이점을 잃으니 기본값은 켜둔다. 스킬을 많이 흡수해 시스템 프롬프트·슬래시 메뉴가 무거워졌을 때만 위 토글로 줄이고, 필요한 스킬만 ~/.omp/agent/skills/에 직접 둔다.
출처 내장문서 config-usage·skills·slash-command-internals·models · ~/.omp/agent/config.yml · omp --help·config list(v16.1.9 실측). 주의 /extensions는 omp 확장 저작 시스템이지 흡수 토글이 아니다.

정직하게 · omp가 (기본) 안 가진 것

omp가 (기본) 안 가진 것

방금 본 대로 기존 환경은 그대로 흡수된다 — 단, 흡수 못 하거나 구조상 없는 것도 정직하게 짚는다.

클라우드 · 원격 실행
CC /ultraplan·/autofix-pr·/background·/batch(워크트리 PR), Codex cloud-tasks. → omp는 로컬 전용, 클라우드 오케스트레이션 없음.
예약 · 스케줄
CC /schedule — 정해진 시각에 작업 실행. omp에는 없음.
호스티드 통합
CC /chrome·/desktop·/mobile·/remote-control·/teleport·/install-github-app·/install-slack-app → Anthropic 호스티드 생태계.
재미 · 온보딩
CC /insights·/powerup·/radio, Codex /pets·/vim.
effort 상한
omp는 effort를 off~xhigh까지 — Anthropic API의 max(xhigh 위)·CC의 ultracode는 미노출(ultracode는 CC 하네스 오케스트레이션). 단 Anthropic 자신이 “Opus는 코딩에 xhigh 권장, max는 frontier 전용·비용 대비 이득 작고 overthinking 위험”이라 명시 → 실손해 미미.
⚠ 단 picker·/model은 omp 최상위(xhigh)를 “max”로 표기(상태줄은 “xhigh”) — 이 “max”는 xhigh의 라벨일 뿐 Anthropic frontier max가 아니다.
omp가 못 하는 건 구조(로컬 전용)에서 온다 — 클라우드·원격·예약·호스티드 통합은 구조상 없다. 반면 스킬·슬래시·플러그인은 갭이 아니다: omp가 ~/.claude·~/.codex에서 읽어 합친다(단 CC 내장 번들 스킬·.claude/agents 에이전트는 제외). 공개 플러그인 스킬은 SKILL.md만 복사하면 네이티브로 동작.
출처 code.claude.com/docs/en/commands · code.claude.com/docs/en/skills(번들=CC 내부) · developers.openai.com/codex · platform.claude.com/docs/en/build-with-claude/effort.

하네스 & 토큰 오버헤드
모든 코딩 AI엔 우리가 못 건드리는 하네스가 있다.
같은 모델이라도 그 하네스가 토큰 소모와 정확도를 가른다.

하네스 = 코딩 AI가 입력 토큰을 관리하고 모델 출력을 실제 파일 변경으로 잇는 인터페이스 층(상위 개념). 모델은 같아도 이 층은 도구마다 다르고 보통 사용자가 못 바꿈.  오버헤드 = 그 하네스가 매 요청에 얹는 토큰 세금(하위 개념). ※ Anthropic은 CLAUDE.md를 ‘메모리(사용자 지침)’, OpenAI는 AGENTS.md를 ‘에이전트 지침’이라 부른다 — 하네스가 읽어들이는 입력이지 하네스 자체가 아니다.

01
하네스 품질(편집 포맷). Hashline로 바꾸면 편집이 한 번에 맞아 재시도가 사라지고 평균 토큰 20~30%↓ — 강한 모델에도 통하는 실질 이득. 정확도 향상은 약한 모델일수록 큼(Grok Code Fast 1 6.7%→68.3%), Sonnet급은 +1~5pp.→ 이 이득은 ‘편집 포맷 품질’에서 온다 — 03의 오버헤드(낭비) 절감과는 다른 레버다.
02
하네스가 변수라는 독립 증거. 같은 Opus가 CORE 벤치에서 자체 기본 하네스 → Claude Code 하네스로 바꾸자 점수 ~2배(제3자 보고).→ 하네스가 점수를 가른다는 일반 증거(여기선 CC 사례) — ‘좋은 하네스가 중요하다’는 omp의 논지와 같은 축이다.
03
오버헤드(낭비)도 실재. 잘 설계된 하네스가 성능을 올려도(02), 같은 하네스가 토큰은 낭비할 수 있다 — 다른 축이다. Claude Code 90일 측정 토큰 ~75%가 오버헤드(규칙·스키마·과거 대화; 상당부분 캐시되나 매 턴 재전송·캐시미스 churn).→ 이 ~75%는 전송 토큰 비중이지 실청구액·지연과 같지 않다 — 캐시 읽기는 할인된다.
출처 README 배지(규모) · blog.can.ac/the-harness-problem + repo packages/typescript-edit-benchmark · x.com/sayashk(Opus CORE, HN #46988596) · acdigest(CC ~75%, 90일) · CC 오버헤드/표시 #44617·#52979 · 캐시미스 churn #40652·#48734. 정직 Hashline=저자 자가벤치(독립검증 0) · Opus CORE=제3자 · CC 75%=1인 90일 · CC 이슈=사용자 보고(thinking-block만 Anthropic 확인). 용어: code.claude.com/docs/memory(CLAUDE.md=메모리) · anthropic.com/engineering/effective-harnesses(하네스 정의).

강점 비교 · 구조적 vs 편의

OMP는 무엇이 다른가

A 구조적 차이 — 설치로는 못 따라옴 (현재 이식 어려움)
1
Hashline 편집 포맷 — 하네스 본체 — OMC·OMX 확장팩으로 이식 불가(호스트 하네스 교체 필요).
2
네이티브 멀티프로바이더(한 세션 40+) — CC=Anthropic·Codex=OpenAI 고정, OMC·OMX는 다른 CLI로 셸아웃. omp는 한 세션서 교체.
3
구조화 서브에이전트 출력(raw JSONL 아님) — 하네스 레벨.
4
컨텍스트 관리 — 의미 기반 무의미결과 제거 — 문서상 동등 기능 못 찾음. (snapcompact 이미지 압축도 구조상 고유하나 한글 본문 많은 세션엔 비활성 → 상세 ④.)
B 편의 차이 — 바로 되느냐 vs 세팅하느냐
항목omp · 코어 내장Claude Code · Codex · 연동/확장
LSP · DAP코어 내장 → 세팅 0IDE 연동(/ide)·외부 의존; DAP는 bash 우회
ripgrep · glob 검색네이티브 내장검색 도구는 있음 — 체감차
에이전트 로스터 · 상호운용 · 관측성기본 제공확장팩·설정으로 가능
격리 프로필--profile·--alias 내장 (인증·세션·설정 완전 격리)CC 네이티브 없음(CLAUDE_CONFIG_DIR 우회)·Codex는 config 프리셋만
진짜 차이는 A(구조적)다. B는 “바로 되느냐 vs 세팅하느냐”의 편의 차이일 뿐 — 남들도 연동·확장으로 상당수 가능.
출처 blog.can.ac/the-harness-problem · oh-my-pi repo · 각 공식 문서. 정직 A만 구조적 강점으로 단언; B는 편의 차이(남도 가능).

상세 ① · Hashline 편집

Hashline 편집

쉽게 — 파일을 통째로 다시 받아쓰지 말고, “17번째 줄, 지문(콘텐츠 해시) a3”처럼 줄번호 + 짧은 지문만 짚어 주는 방식. 지문이 안 맞으면(=그새 누가 바꿈) 자동 거부 → 엉뚱한 줄을 안 건드린다.
개발자용
read/search가 줄마다 22:f1| ...처럼 콘텐츠 해시를 부여, 편집은 앵커 참조 → 옛 내용 재현 불필요. CC str_replace는 공백까지 완전일치 요구(“String not found” 다발), Codex apply_patch는 전용 diff라 타모델 실패율↑(Grok 4 50.7%, GLM-4.7 46.2%).
벤치
평균 토큰 20~30%↓ · Claude Sonnet 4.5 +1.6pp · Gemini 3 Flash +5pp(→78.3%) · Grok 4 Fast 출력토큰 −61% · MiniMax 2.1배. 정확도 향상은 약한 모델일수록 큼 — 극단값 Grok Code Fast 1 6.7%→68.3%. (16모델·180과제×3)

상세 ③ · IDE · 디버거

IDE · 디버거 코어 내장

LSP
정의 · 참조 · 이름변경 · 진단 · code_actions
omp는 클라이언트 코어 내장 → 연동 세팅 0. CC·Codex는 /ide 연동·외부 의존으로 비슷하게 가능(편의 차이).
DAP
브레이크포인트 · 스텝 · 스택 · 변수
omp는 디버거를 구조화 도구로 내장 → 연동 세팅 0. 경쟁자는 native 디버거 도구면이 없어 bash 우회(편의 차이 중 체감 큰 쪽).
정리 — 분류는 '편의'지만 DAP는 체감 큰 제품 차이 — 경쟁 에이전트엔 구조화 디버거 도구면(브레이크포인트·스텝·변수)이 아예 없다. omp는 클라이언트 코어 내장이라 연동 세팅 0(언어 서버·디버그 어댑터 설치는 도구 공통).

상세 ④ · 컨텍스트 관리 (정직)

컨텍스트 관리

솔직히 — omp 압축은 ① 더 큰 컨텍스트 모델로 승격 · ② 무의미·과대 출력 정리 · ③ LLM 요약 3단계인데, 대부분 CC·Codex와 비슷하다. omp 고유는 둘 — 하나(의미 필터)는 견고, 하나(snapcompact)는 한글 세션엔 조건부.

공통 — 차별 아님
· ③ LLM 요약 = Claude Code와 사실상 동일 (Codex는 서버측 blob)
· ② 크기·오래된 출력 정리(오래된·큰 것부터 지움)도 CC(microcompact)·Codex(trim)가 함

사용량 한도 표시

omp가 보여주는 한도는 두 가지 — 세션 한도(5시간 롤링)와 주간 한도(7일). 상태줄 usage 바가 둘 다(활성 provider 기준) 표시하고, omp usage로 정확 수치·리셋 시각 확인. (Anthropic 구독 기준)

즉석 확인
· /usage — 세션 안에서 바로
· omp usage — 모든 인증 계정 자동. 필터: omp usage -p anthropic
· 추세: omp usage --history --days N
상태줄 usage 바
현재 활성 모델의 provider 한도를 자동 추적해 표시(세션·주간 둘 다). 전체 계정은 위 명령으로.
~/.omp/agent/config.yml
statusLine:
  preset: custom
  leftSegments: [model, mode, path, git,
                 context_pct, cost, usage]
  rightSegments: [session_name]
주의 ① preset: custom이어야 커스텀 세그먼트가 적용된다.
주의 ② 세그먼트 id는 snake_case — context_pct·session_name·usage. 적용은 omp 재시작.
PROMPT · 직접 고치기 애매하면 그대로 붙여넣기
내 config.yml 상태줄에 usage·context_pct·cost 세그먼트를 추가하고 preset을 custom으로 바꿔줘.

Advisor — 매 턴 지켜보는 2차 모델

세션에 붙는 선택적 2차 모델. 매 턴이 끝나면 트랜스크립트를 리뷰하고 read-only로 워크스페이스를 살펴 간결한 조언을 1차 세션에 주입한다 — 1차 모델이 지나칠 실수·놓친 점을 독립 시점에서 잡아준다.

켜기
영속: config.yml → advisor.enabled: true · 세션: /advisor(on/off/status/dump) · 시작: --advisor. 쓰는 모델은 modelRoles.advisor로 지정. 전용 단축키는 없다(액션 ID 미등록).
튜닝
advisor.immuneTurns(초반 N턴 면제)·syncBacklog·subagents로 개입 빈도·범위 조절.
효과
도구 오용·사실 오판·누락을 매 턴 교차검증해 짚어준다. 1차 세션과 다른 시점이라 자기 점검이 놓치는 걸 잡는다.
⚠ 비용 (정직)
매 턴 2차 모델을 추가 호출 → 그만큼 토큰이 더 든다. “감시 비용을 내고 품질을 산다”는 트레이드오프 — 고위험·장기 작업일수록 가성비가 크다.
출처 내장문서 advisor-watchdog · omp config(advisor.enabled·immuneTurns·syncBacklog·modelRoles.advisor) · CLI --advisor·세션 /advisor. 주의 2차 모델 토큰 비용 발생.
